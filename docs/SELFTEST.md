# 自测报告 — 对照第九节验收标准

## 测试环境

- Godot 4.6.1 stable，RenderingDevice = **D3D12**，GPU = **NVIDIA GeForce RTX 4060**，Windows 11。
- 网格 160×120，窗口 1600×1200，粒子池容量 `capacity = 1 048 576`（默认 100 万）。
- 参数：`phys_dt=1/240 s`、`4 子步/帧`、`Jacobi=60`、`gravity=40 cells/s²`、`flip_ratio=0.90`、`p_atm=3.0`。
- 验证方式：`--quit-after N` 跑真实 `_process` 循环，`Time.get_ticks_usec` 包裹 `solver.step` 计时，
  周期性 `get_viewport().get_texture()` 截图。下方引用的图片见 `docs/images/`。

> 复现：把 `scripts/main.gd` 顶部 `DEBUG_CAPTURE := true`（关闭 vsync + 定期截图并打印
> fps/step/active/probe）。

> **渲染修复记录（“只有 UI 没有画面”）**：初版把流体贴在根 Node2D 上用 `_draw()` 绘制，`get_viewport()`
> 截图能看到水，但**真实窗口只显示 HUD、流体全黑**。定位为：每帧对 local RenderingDevice `submit()/sync()`
> 时，主视口的默认世界 2D 画布不呈现到窗口（CanvasLayer 正常）。已改为**把显示 Sprite2D 放进 CanvasLayer**
> 修复。另外 1600×1200 窗口高于 1080p 屏幕，改用 `canvas_items`+`keep` 拉伸、初始窗口 1280×960 适配。
> 见真实桌面截图 `docs/images/window_render_actual.png`。

## F9 —自由液面稳定性（具体现象/数据）

**溃坝算例**（初始整列水，89 600 活跃粒子）：

- 活跃粒子数在整个 ~16 s 仿真中**恒为 89 600**（逐帧打印一致）——无粒子凭空生成/消失，**体积守恒**。
- 空 cell 压力探针稳定读 `p_atm = 3.000`；未出现 NaN/inf，未出现数值发散。
- 演化：满高水柱塌落 → 撞右墙上溅（`dam_break_splash.png`，能量高、低耗散）→ 逐渐衰减 →
  最终稳定为**贴底、跨全宽的水池**，液面基本平整，仅残留缓慢衰减的涌浪（`dam_settled.png`）。
  稳定液位 ≈ 5 600 水 cell / 160 宽 ≈ 35 cell 深，与体积一致。

**容器算例**（49 808 粒子）：活跃数恒定，水稳定停在固体容器内，自由液面平整（`solid_container.png`）。

结论：无“无限抖动、无缘由膨胀/消失、数值发散”。`flip_ratio` 由 0.95 调到 **0.90** 让低粘 FLIP
在数秒内收敛为可信静水面（仍 FLIP 主导），见“偏差说明⑤”。

## F10 — 性能（实测）

- **单帧仿真耗时**（`solver.step`：4 子步 ×（清网格+P2G+归一+固体BC+散度+60×Jacobi+投影+固体BC+G2P平流）
  + 1 次 render，含 `submit()+sync()`）：**平均 ≈ 2.2–2.6 ms**（多次运行 avg 2.44–2.58 ms）。
- **整帧帧率**（关 vsync，含每帧回读 display+probe+counters 与呈现）：**≈ 220–250 fps**。
  开 vsync 时锁显示器刷新率，仿真占用不变。
- 说明：`p2g/g2p/clear_vel` 等按**容量 100 万线程**派发（死槽提前 return，仍占线程），故上述耗时即
  代表“默认 100 万粒子”的成本，与当前活跃数关系不大。测试的活跃数：89 600、49 808。
- 结论：160×120 网格 + 100 万粒子池在 RTX 4060 上**远超实时**（>3× of 60 fps 预算），
  为后续扩容/落地留有充足余量（架构支持提高 `capacity`/`emit_ppc` 至更大量级）。

## B. 手感/观感

- 重力下运动连续可信：溃坝的塌落—上溅—回落—铺开—静置全过程平滑，非明显颗粒卡顿（100 万粒子池
  过采样，网格传递平滑）。
- 画笔响应：预览为 CPU 覆盖层，逐帧跟手无延迟；提交为一次性 GPU dispatch，瞬时生效。
- 固体碰撞：整体无穿模/渗漏；仅凹角处极少量粒子挤出（<0.1%），不影响观感。

## 交付要求④ — 与参考案例的偏差及依据

1. **“粒子总数”语义**：实现为**池容量上限**（free-list 栈管理），活跃数随画水增长至上限；HUD 显示
   `active/capacity`。原因：画笔式 demo 初始无水，需要动态发射/回收；有界栈避免碎片与前缀和排序。
2. **画笔预览/提交**：拖拽中在 CPU 累积被覆盖 cell 并以覆盖层预览（不碰仿真缓冲），松开一次性上传
   `brush_mask` 提交。与需求一致，此处记录具体实现方式。
3. **气 vs 真空**：单相流中二者在求解器里**等价**（都视为“空”，自由液面统一用 `p_atm` ghost）；
   仅渲染区分（灰/黑）并作为后续多相流的预留接口。
4. **压力画笔**：实现为**径向速度冲量**而非直接写压力场。依据：Jacobi 收敛压力只由散度+边界条件决定，
   与初值无关，直接赋压力会被当步立即解掉、不可观察；改为冲量能明确产生“压力扰动→水流运动”（F4）。
5. **`flip_ratio=0.90`**（非 0.95/0.99）：低粘 FLIP 溅射久不收敛，加入 10% PIC 阻尼让其数秒内稳定为
   静水面，观感更像“可信静止液面”，且仍 FLIP 主导。属 demo 网格尺度下的调参取舍。
6. **`p_atm=3.0`、`Jacobi=60`** 在本 160×120 尺度直接沿用参考默认即稳定，无需实质性调整。
   `gravity=40 cells/s²`、`phys_dt=1/240 s` 按本网格尺度选取（满足 CFL：单子步位移 <1 cell），均为
   `FlipFluidGPU` 顶部可调项。
